- Yesterday May 1 we shipped UW options migration thinking that resolved
  Pipeline B's DNS thread exhaustion crashes
- This morning May 2, Pipeline B crashed AGAIN at GEV (ticker 47/125)
- Same crash signature: yfinance ^VIX/ES=F getaddrinfo() thread failures
- Root cause was incomplete diagnosis yesterday: we eliminated yfinance
  from options chain, but missed per-ticker macro fetches

What was missed yesterday:
- features/builder.py:_market_return calls mc.download for 10 macro symbols
- mc.download routes unsupported symbols (^VIX, ES=F) to yfinance fallback
- Each ticker triggers ~10 macro fetches; ~46 tickers exhaust DNS pool

- _get_macro_cached() helper wraps mc.download with cache lookup
- _market_return uses cache (covers SPY, XLK, ^VIX, ES=F, USO, SMH,
  IGV, LQD, HYG)

  which is too short for 30+ min Pipeline B run)
- 1,250 yfinance calls per run -> ~10 yfinance calls per run

Validation:
- 5-ticker unit test: cache fills with 10 entries on first ticker, holds
  steady on subsequent. GEV (this morning's crash victim) builds in 7.9s.
- 54-ticker integration test: 162 models trained, 0 failed
- Build times: 14.9s first ticker, 7.9-11.8s subsequent

Status:
- Commit 5458524 pushed to research-track
- builder.py.bak.before_macro_cache backup in place
- Models from this morning's partial run (46 tickers A-FVRR) intact
- Models from yesterday's full retrain (all 125) intact

Pipeline B Monday May 4 07:00 Vietnam:
- Should complete cleanly on all 125 tickers
- 14-day observation window starts Monday
- If crashes recur, root cause is something other than macros — would
  need to investigate massive_client.py:189, :336 yfinance call sites

Pattern noted for future debugging:
- Yesterday's "fix" was correct but incomplete
- DNS thread exhaustion can be triggered from MULTIPLE call paths

  thread pool at 125-ticker scale, even if individual call seems light
